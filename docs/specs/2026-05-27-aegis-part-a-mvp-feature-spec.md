# Feature Spec: Part A (MVP) — Single Agent & Core Workbench

## Summary
The Part A MVP provides a web-based workbench where a user can upload a health insurance denial letter and receive a drafted appeal, generated by a single ADK agent utilizing 7 tools. The backend is monitored entirely by Phoenix Cloud, with Phoenix MCP providing runtime introspection.

## Problem
Users facing health insurance denials lack the expertise to draft appeals. Insurers use automated systems to deny claims, creating an asymmetry that results in 99% of denied claims being abandoned. 

## User Scenarios
- **US-1 (Patient):** A patient pastes their denial letter into the UI. The system parses it, looks up relevant laws/policies, and drafts an appeal letter within 60 seconds.
- **US-2 (Reviewer):** A human reviewer can see the generated draft alongside exact citations from the local corpus to verify the agent's claims.

## Functional Requirements
- **FR-1:** The frontend MUST allow text input of a denial letter.
- **FR-2:** The backend MUST route the input to a single Google ADK agent.
- **FR-3:** The agent MUST have access to 7 tools (`case_parser`, `corpus_retrieval`, `phoenix_mcp_lookup`, `playbook_loader`, `drafter`, `self_check`, `simulator`).
- **FR-4:** The UI MUST display the drafted appeal and a confidence/citation panel side-by-side.
- **FR-5:** The system MUST evaluate the drafted appeal against the 12-case benchmark (6 train, 6 holdout).

## Non-Functional Requirements
- **NFR-1:** The UI motion budget MUST respect Constitution C-3.1 (400ms default).
- **NFR-2:** All backend traces MUST be emitted to Phoenix Cloud (Constitution C-5.1).
- **NFR-3:** The system MUST NOT process any real Protected Health Information (Constitution C-2.1).

## Acceptance Criteria
- **AC-FR-1.1:** Given a user is on the workbench, When they paste a synthetic denial letter and click submit, Then the frontend displays a loading state and sends the payload to the backend.
- **AC-FR-2.1:** Given a backend request, When the ADK agent finishes drafting, Then a structured JSON response containing the draft and citations is returned.
- **AC-FR-5.1:** Given a run on the 12-case benchmark, When evaluated by the LLM Judges, Then the system MUST pass the J1 and J2 hard safety gates with 100% success on holdout cases.

## Edge Cases
1. The user inputs a denial letter that is completely unrelated to health insurance (e.g., auto insurance).
2. The `corpus_retrieval` tool fails to find any relevant laws.
3. The LLM API (Gemini) rate limits or times out during the drafting phase.

## Out of Scope
- Multi-agent swarm (deferred to Part B).
- Autonomous self-improvement loop (deferred to Part B).
- Live filing of appeals with insurers.
- Medical or legal advice.
